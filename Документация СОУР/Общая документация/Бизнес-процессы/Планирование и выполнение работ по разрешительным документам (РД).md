## 1. Цель процесса
Обеспечить полный цифровой цикл: от получения планового заказа ТОРО из SAP (СУПА) до закрытия разрешительного документа в СОУР и автоматической передачи фактических данных о трудозатратах, объемах и материалах обратно в СУПА и 1С:ЗУП.

## 2. Участники и роли

| Роль                                         | Ответственность                                                                                                                                                           |
| :------------------------------------------- | :------------------------------------------------------------------------------------------------------------------------------------------------------------------------ |
| Инженер ПТГ                                  | Получает заказы из SAP на месяц, формирует задания, создает РД в статусе «Планирование» и распределяет их по датам и мастерам.                                            |
| Мастер / Инженер РЭС                         | Оформляет РД в Вебе: прикрепляет заказы ТОРО, назначает бригаду (из справочника или шаблона). Может выдавать РД без участия диспетчера для определенных типов документов. |
| Диспетчер ОДС (Филиал/ИА)                    | Проверяет наряды-допуски. Переводит РД в статус «Проверен», подтверждая правильность мер безопасности.                                                                    |
| Выдающий наряд                               | Присоединяет ЭЦП, переводя РД из статуса «Проверен» в «Выдан». Часто эту роль совмещает Мастер.                                                                           |
| Производитель (ПР) / Отв. руководитель (ОРР) | Принимает РД в работу на МУ, фиксирует перемещения, выполнение технологических этапов (с фото/видео), дефекты и фактические объемы.                                       |
| Инженер (Веб) / Мастер                       | В веб-интерфейсе проверяет синхронизированный с МУ факт, корректирует интервалы трудозатрат и инициирует отправку данных в СУПА.                                          |

## 3. Ключевые особенности процесса

- **«Одна кнопка» для простых РД:** Для типов Распоряжение, Поручение и Работы по перечню этап проверки диспетчером может быть пропущен. Система автоматически проходит цепочку «Планирование» → «На проверке» → «Проверен» → «Выдан» при нажатии кнопки «Выдать».
- **Валидация занятости:** РД нельзя взять в работу на МУ, если хотя бы один член бригады уже задействован в другом активном РД (статус «В работе» или «Перерыв»).
- **Фиксация перемещений:** Время доставки до объекта фиксируется нажатием кнопок на МУ. Если перемещения не фиксировались, время доставки обнулится, и трудозатраты придется корректировать вручную в Вебе.
- **Локальное выполнение:** Для подрядчиков или при отсутствии связи возможен режим работы без МУ, где все данные (включая временные интервалы) вносятся мастером вручную в веб-интерфейсе.

## 4. Схема процесса 

![[Pasted image 20260515151321.png]]

```plantUML
@startuml
skinparam handwritten false
actor "Инженер ПТГ" as ptg
actor "Мастер (Веб)" as master
actor "Диспетчер ОДС" as disp
actor "ПР/ОРР (МУ)" as field
actor "Инженер/Мастер (Веб)" as eng_confirm

== Планирование ==
ptg -> ptg : Получает заказы из SAP
ptg -> ptg : Создает задание на период
ptg -> ptg : Создает РД в статусе "Планирование"

== Оформление и выдача ==
master -> master : Заполняет РД (привязка ТМ из справочника)
master -> master : Прикрепляет заказы ТОРО (только Released)
master -> master : Назначает бригаду (проверка прав)

alt Тип РД = Наряд-допуск
    master -> master : Передает на проверку (статус "Передан на проверку")
    disp -> disp : Проверяет РД (статус "Проверен")
    master -> master : Выдает РД с ЭЦП (статус "Выдан")
else Тип РД = Распоряжение/Поручение/РпП
    master -> master : Нажимает "Выдать" (авто-переход в "Выдан")
end

== Выполнение (Поле) ==
field -> field : Берет в работу (статус "В работе", фиксация координат)
field -> field : Выполняет этапы (фото/видео, чекап)
field -> field : Фиксирует факт (объемы, материалы, дефекты)
field -> field : Завершает работы (статус "Подтверждение работ")

== Закрытие ==
field -> eng_confirm : Синхронизация данных
eng_confirm -> eng_confirm : Проверяет факт, корректирует Ч/Ч на вкладке "Интервалы"
eng_confirm -> eng_confirm : Нажимает "Передать в СУПА" (статус "Отправлен в СУПА")
eng_confirm -> eng_confirm : Получает подтверждение от SAP и 1С:ЗУП (статус "Закрыт")
@enduml
```
