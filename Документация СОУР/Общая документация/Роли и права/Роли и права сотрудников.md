## Основные роли

| Роль                      | Код роли в БД                     | Описание                                                                                                                                                                                                         |
| :------------------------ | :-------------------------------- | :--------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- |
| Мастер/инженер РЭС        | SOUR_MASTER_ХХХХ_YYY              | Планирует работы, создаёт и выдаёт РД в рамках своего РЭС. Имеет доступ к веб-интерфейсу и МУ. Видит РД, где он автор, член бригады или где участвуют сотрудники из его шаблона бригады (visibility_my_brigade). |
| Исполнитель               | SOUR_WORKER_ХХХХ_YYY              | Выполняет работы по РД с мобильного устройства. Имеет доступ к МУ и ограниченный доступ к вебу. Видит только те РД, в которые он назначен в состав бригады (visibility_my_rd).                                   |
| Диспетчер ОДС (Филиал/ИА) | SOUR_ODS_XXXX / SOUR_ODS_IA       | Проверяет правильность заполнения нарядов-допусков и подтверждает их выдачу с помощью ЭЦП. В большинстве случаев не имеет доступа к МУ (доступ только через веб).                                                |
| Администратор (ИА)        | SOUR_ADMIN                        | Обладает максимальным уровнем прав (CRUD) на все объекты системы по всем филиалам. Управляет привязкой учетных записей AD к сотрудникам и настройкой заказов ТОРО.                                               |
| Эксперт (Филиал/ИА)       | SOUR_EXPERT_XXXX / SOUR_EXPERT_IA | Создаёт любые объекты системы (кроме справочников пользователей). Имеет доступ к вебу и МУ с правом создания заданий.                                                                                            |

## Специальные роли

| Роль | Код роли в БД | Описание |
| :--- | :--- | :--- |
| Сторонний персонал (Филиал) | CONTRACT_WORKER_ХХХХ | Сотрудники подрядных организаций. Имеют доступ только к МУ (для выполнения РД в поле). В веб-интерфейсе для них предусмотрен режим «Локального выполнения», где факт работ за них вносит сотрудник ДЗО. |
| Сотрудник охраны труда | SOUR_OHR_TRUDA_XXXX | Заполняет справочники по персоналу (права по РД, допуски, группы ЭБ) в рамках своего филиала. Не имеет доступа к МУ. |
| Специалист (просмотр) | SOUR_TECHN_XXXX | Имеет права только на чтение (просмотр) объектов системы и доступ к аналитическим дашбордам. Доступ к МУ отсутствует. |
## Матрица уровней доступа (CRUD):

Для каждого объекта в БД аналитику необходимо учитывать системный уровень доступа (`public.system_right`):
- **Запрет** (`forbidden`): Объект скрыт от пользователя.
- **Чтение** (`to_read`): Только просмотр данных.
- **Редактирование** (`to_read_update`): Изменение существующих записей.
- **Создание** (`to_read_update_create`): Добавление новых записей.
- **Удаление** (`to_read_update_create_delete`): Максимальный уровень (доступен Администраторам и Экспертам).
## Технические нюансы и бизнес-логика:

- - **Принцип суммирования прав:** Если пользователю назначено несколько ролей (например, Мастер и Эксперт НСИ), система выбирает **максимальный уровень доступа** из доступных для конкретного объекта.
- **Сквозная авторизация:** СОУР не хранит пароли. Аутентификация идет через **Active Directory (AD)**. Сопоставление учетки AD и сотрудника делается вручную администратором в карточке `employee`.
- **Ограничение сессий:** Одна учётная запись — **строго одно мобильное устройство**. Одновременный вход под одним логином с двух МУ приводит к немедленной потере данных и ошибкам синхронизации РД.
- **Видимость РД (Additional Access):**
    - `visibility_my_rd`: Сотрудник видит только РД, где он в составе бригады.
    - `visibility_my_brigade`: Мастер видит РД своих сотрудников из шаблона штатной бригады.
- **Синхронизация ролей:** Роли назначаются во внешней системе **ДИР (Service Desk)**. Техподдержка СОУР не имеет прав на назначение ролей пользователям.
- **Конфликт избыточных ролей:** Если у пользователя более 3–4 ролей, мобильное приложение может выдавать ошибку при первой инициализации из-за слишком большого пакета данных.
