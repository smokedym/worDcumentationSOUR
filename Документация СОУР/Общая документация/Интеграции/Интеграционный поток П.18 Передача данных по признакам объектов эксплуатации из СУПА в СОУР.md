## Параметры потока

| Параметр      | Значение                                                                    |
| :------------ | :-------------------------------------------------------------------------- |
| Направление   | СУПА (SAP) → СОУР                                                           |
| Протокол      | SOAP/XML                                                                    |
| Транспорт     | ЕИП (Единая интеграционная платформа)                                       |
| Триггер       | Периодический запрос ЕИП по регламенту или ручной запрос из интерфейса СОУР |
| Периодичность | Еженедельно (автоматически) / по запросу (вручную)                          |
| Обработка     | Запись в таблицу `measurement_attributes`                                   |
| Маршрутизация | По HTTP-заголовку `source_platform` (center или cip)                        |

## Общая информация

- **Назначение:** Передача значений технических признаков (атрибутов) для технических мест (ТМ) и единиц оборудования (ЕО) из мастер-системы SAP.
- **Инициатор:** Инициатором выступает ЕИП. По регламенту (раз в неделю) формируется запрос на основании выборки всех объектов эксплуатации (ТМ и ЕО), хранящихся в СОУР.
- **Особенность обработки:** Если при вставке признака в потоке П.23 (точки измерений) имеется ссылка на данный признак (`measurement_attribute_id`), то данные также дублируются в таблицу `measurement_points`.

## Какие данные передаются

### Запрос (СОУР → СУПА)

| Атрибут запроса | Описание | Источник данных |
| :--- | :--- | :--- |
| `IT_ASSETID` | Код объекта (ТМ или ЕО) в SAP | `equipments.integration_entity_id` или `technical_places.integration_entity_id` |
| `IT_ATTRID` | Имя признака (опционально) | — |

### Ответ (СУПА → СОУР)

| Атрибут XML (Центр/ЦиП) | Поле БД СОУР | Описание |
| :--- | :--- | :--- |
| `ASSETID` | `equipments_id` / `technical_place_id` | Внутренний ID объекта в СОУР, определенный по коду SAP |
| `ATTRID` | `measurement_points_attributes_id` | Ссылка на справочник атрибутов |
| `ATTRNAME` | `name` | Наименование признака |
| `VALUE_CHAR` | `value_char` | Строковое значение признака |
| `VALUE_FROM` | `value_from` | Числовое значение «От» |
| `VALUE_TO` | `value_to` | Числовое значение «До» |
| `MUNIT` | `measurement_unit_id` | Ссылка на справочник единиц измерения |

## Ключевые правила обработки

### 1. Определение объекта и признака

- Система сопоставляет `ASSETID` из сообщения с `integration_entity_id` в таблицах `equipments` или `technical_places` с учетом `source_platform`.
- Код признака (`ATTRID`) и единица измерения (`MUNIT`) ищутся в соответствующих справочниках СОУР по их интеграционным идентификаторам.

### 2. Логика «Опасных мест» для сегмента ЦиП (source_platform = cip)
Это критически важное правило, которое должно быть описано в потоке П.18:

- **ЕСЛИ** приходит признак с `integration_entity_id` = 'ZPM_TRAVMOOPASNOE_MESTO' **И** значение `VALUE_CHAR` = 'Да', **ТОГДА** этот объект автоматически помечается в СОУР как **Опасное место**.
- Данные записываются в таблицу `dangerous_places`, где в поле `constraint` (ограничение) вносится текст **«Травмоопасное место»**.
- Если позже придет значение «Нет» или пустое поле, информация об опасном месте может быть удалена (при условии отсутствия подтвержденных дефектов по этому объекту).

### 3. Связь с точками измерений (П.23)

- Если на вставляемый признак есть ссылка из потока точек измерений (П.23), данные записываются в таблицу `measurement_points`.
- Заполняются поля: `technical_place_id` / `equipments_id`, `value_char`, `value_from`, `value_to`, `measurement_unit_id`.

### 4. Валидация и ошибки

- В случае неуспешной валидации данных на стороне ЕИП, системе СОУР передается ответ с ошибкой.
- Если СУПА недоступна, инициируется логика повторной отправки (RETRY).
